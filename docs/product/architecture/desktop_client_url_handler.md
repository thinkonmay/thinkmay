# Desktop client custom URL handler

This document describes the `thinkmay:` custom URL scheme used to launch the native Go desktop client from the website or operating system shell.

The desktop client architecture is documented separately in `docs/desktop_client_architecture.md`. The important link for this document is that the desktop client can be configured entirely from a remote URL: the URL carries the VM ID, listener tokens, codec, transport address, and runtime options that the Go client maps into direct QUIC connections.

## Purpose

The custom handler lets a browser page or desktop environment open an installed native client with a single launch URL instead of asking the user to copy CLI flags.

```text
thinkmay:https://thinkmay.net/en/remote/?vmid=<vm-id>&video=<video-listener>&audio=<audio-listener>&data=<hid-listener>&mic=<mic-listener>&codec=h265
```

The handler is only a launch mechanism. It does not authenticate the user by itself. The security-sensitive values are still the short-lived listener IDs generated by the backend session flow.

## End-to-end launch flow

```mermaid
flowchart LR
    Website["Website remote page"] --> Link["thinkmay: remote URL"]
    Link --> OS["OS URL scheme handler"]
    OS --> Client["thinkmay-client"]
    Client --> Parser["client/config.Parse"]
    Parser --> QUIC["QUIC FinalTarget streams"]
    QUIC --> Proxy["Thinkmay proxy"]
```

1. The web app starts or resumes a VM session and receives listener IDs for video, audio, HID/data, microphone, and logs through the existing PocketBase/proxy session flow.
2. The web app builds a normal remote page URL containing listener query parameters.
3. The URL is wrapped with the `thinkmay:` scheme.
4. The operating system dispatches the URL to the registered Thinkmay desktop client.
5. The client passes the raw URL to `config.Parse`, which strips the custom scheme and maps query parameters into `client/config.Config`.
6. The desktop app starts direct QUIC streams to the proxy using `{vmid, listenerID}` instead of using WebRTC signaling.

## URL grammar

The parser accepts both of these forms:

```text
thinkmay:https://thinkmay.net/en/remote/?vmid=<vm-id>&video=<token>
thinkmay://https://thinkmay.net/en/remote/?vmid=<vm-id>&video=<token>
```

Internally, the parser removes the `thinkmay:` prefix and trims leading `/` characters before parsing the remainder as the real remote URL.

The URL after the prefix must be an absolute URL with a host. The host is used as the default QUIC address when no explicit `addr` or `server` query parameter is present.

## Query parameters

| Query parameter | Config field | Required | Notes |
| --- | --- | --- | --- |
| `vmid` | `VMID` | Yes | Target VM ID used in the QUIC `FinalTarget` handshake. |
| `video` or `token` | `Token` | Yes | Video listener ID. `video` is the preferred desktop-link key; `token` is also accepted. |
| `audio` or `audio-token` | `AudioToken` | No | Enables audio playback when present. |
| `mic` or `mic-token` | `MicToken` | No | Enables microphone capture when present. |
| `data` or `data-token` | `DataToken` | No | Enables HID input and is required for USB forwarding. |
| `codec` | `Codec` | No | `h264`, `h265`/`hevc`, or `av1`; defaults to `h264`. |
| `addr` | `Addr` | No | Explicit QUIC host/port. If the port is omitted, port `50005` is added. |
| `server` | `Addr` | No | Alternative explicit QUIC host/port, also defaulting to port `50005`. |
| URL host | `Addr` | No | Used when `addr` and `server` are absent; default port is `443`. |
| `hwaccel` | `HWAccel` | No | Overrides decoder hardware selection. |
| `present` | `Present` | No | Overrides presenter selection. |
| `width`, `height` | `Width`, `Height` | No | Initial window size. |
| `fullscreen` | `Fullscreen` | No | Boolean; empty value means `true`. |
| `vsync` | `VSync` | No | Boolean; empty value means `true`. |
| `fps` | `Fps` | No | Initial remote framerate control. |
| `bitrate` | `Bitrate` | No | Initial remote bitrate control. |
| `stats` | `Stats` | No | Boolean terminal metrics toggle. |
| `usb` | `USBForward` | No | Enables USB forwarding; requires `data`. |
| `usb-list` | `USBList` | No | Lists local USB devices and exits. |
| `usb-all` | `USBAll` | No | Forwards all visible USB devices when USB forwarding is enabled. |
| `usb-vidpid` | `USBVIDPID` | No | Comma-separated `vid:pid` allowlist. |
| `usb-detach-kernel-driver` | `USBDetachKernelDriver` | No | Allows USB forwarding to detach kernel drivers for allowlisted devices. |

Explicit CLI flags always override values from the URL. For example, launching `thinkmay-client -url <url> -codec h264` keeps `h264` even if the URL contains `codec=h265`.

## Platform registration

### Windows

The Windows client installer registers the protocol under the current user hive:

```text
HKCU\Software\Classes\thinkmay
HKCU\Software\Classes\thinkmay\shell\open\command
```

The command is:

```text
"$INSTDIR\thinkmay-client.exe" -url "%1"
```

The installer writes the key during install and deletes `HKCU\Software\Classes\thinkmay` during uninstall. The client package workflow verifies both registration and cleanup.

### macOS

The macOS app bundle declares the URL scheme in `Info.plist`:

```xml
<key>CFBundleURLSchemes</key>
<array>
  <string>thinkmay</string>
</array>
```

The Go entrypoint registers a Cocoa AppleEvent handler for `kAEGetURL`. During cold start, if there are no CLI arguments, the app pumps the macOS run loop briefly and uses the received URL as the config input.

### Linux

The Linux package ships a desktop entry with:

```ini
Exec=thinkmay-client -url %u
MimeType=x-scheme-handler/thinkmay;
```

The `.deb` package installs the desktop entry under `/usr/share/applications` and refreshes the desktop database. The tarball includes the same `.desktop` file and a README instructing users to run:

```bash
xdg-mime default thinkmay-client.desktop x-scheme-handler/thinkmay
update-desktop-database ~/.local/share/applications/
```

## Runtime behavior

The URL handler enters the same config path as explicit CLI usage. After parsing, normal startup continues:

1. Validate required `addr`, `vmid`, and video token fields.
2. Normalize codec and presentation settings.
3. Create the mandatory video QUIC client.
4. Create optional audio, microphone, and HID/data QUIC clients when their tokens are present.
5. Start decode, presentation, input, audio, and control loops.

The custom scheme does not create a separate protocol at the transport layer. It only bootstraps the native client into the existing desktop QUIC architecture.

## Validation coverage

Current automated checks cover the handler at three layers:

| Layer | Coverage |
| --- | --- |
| Parser | Unit tests parse both `thinkmay:` and `thinkmay://` URL forms. |
| Windows packaging | CI silently installs the NSIS package, checks registry values, runs uninstall, and verifies cleanup. |
| macOS packaging | CI checks the app bundle `Info.plist` contains the `thinkmay` URL scheme. |
| Linux packaging | CI checks the `.desktop` file contains `x-scheme-handler/thinkmay` and `-url %u`. |

## Caveats

- Listener IDs in the URL are bearer credentials for the streaming session. Do not log full launch URLs in user-visible or long-lived telemetry.
- The macOS AppleEvent path is currently a startup handoff. The app only polls for a URL before parsing config; it does not yet support switching sessions through a second URL while an existing client process is already running.
- Linux tarball users may need to register the `.desktop` file manually with `xdg-mime`; the packaged `.deb` installs the desktop entry but desktop-environment defaults can still vary.
- The older Windows runtime bundle documented in `windows_bundle.md` also registers `thinkmay://`, but it points at `daemon.exe`. The dedicated desktop client installer points at `thinkmay-client.exe -url "%1"` and is the current native-client handler path.
